Using Eclipse 4.16M2 (Build id: 20200507-0346), start with the following code. public class InliningProblem { public static void main(String[] args) { var i_S = i_s(); System.out.println(i_S.i + i_S.s); } static record I_S(int i, String s) {} private static I_S i_s() { return new I_S(1, "abc"); } } Using Inline Method on the i_s() invocation turns the code into the following. import InliningProblem.I_S; public class InliningProblem { public static void main(String[] args) { var i_S = new I_S(1, "abc"); System.out.println(i_S.i + i_S.s); } static record I_S(int i, String s) {} } This does not compile, as the import is invalid. Inline Method has three options (inline "All invocations" with or without "Delete method declaration", and inlining "Only the selected invocation"), the same import is inserted no matter which one is chosen. I've set the bug's severity as major, as it's very important that refactorings' results be conservative enough not to produce invalid code.

https://docs.oracle.com/javase/specs/jls/se7/html/jls-7.html#jls-7.5 clearly specifies that A type in an unnamed package (§7.4.2) has no canonical name, so the requirement for a canonical name in every kind of import declaration implies that (a) types in an unnamed package cannot be imported, and (b) static members of types in an unnamed package cannot be imported. As such, §7.5.1, §7.5.2, §7.5.3, and §7.5.4 all require a compile-time error on any attempt to import a type (or static member thereof) in an unnamed package. This bug is reproducible only if the class is created in default package in which imports do not work as he java spec specified above. So, Reducing the severity of this bug,as this issue does not occur in named packages.
