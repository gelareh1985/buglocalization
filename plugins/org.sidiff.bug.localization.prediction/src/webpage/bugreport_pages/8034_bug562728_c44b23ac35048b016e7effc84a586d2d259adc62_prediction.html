The test below prints "default" and then "3b <- WTH?" when run in Eclipse. When compiled and run with OpenJDK 13 (with --enable-preview) or 14 it correctly only prints "default". public class Test { static public void main (String[] args) { int a = 0x21; int b = 0xff; switch (a) { case 0x21 -> { switch (b) { default -> System.out.println("default"); } } case 0x3b -> System.out.println("3b <- WTH?"); } } } Variations: 1) "default" can be replaced with "case 0xff" or "case 0" and the behavior is the same. 2) Adding a statement block (curly braces) after the default arrow prevents the fall-through. 3) Adding other cases does not prevent the fall-through unless they are a) after the executed case, and b) they have a statement block (curly braces). 4) Adding any statement after the inner switch, at the end of the 0x21 block, prevents the fall-through.

Ok, I made some investigations. I'll try to explain my findings. Let's call switch1 the enclosing switch statement (switch on a) and switch2 the inner switch. public class Test { static public void main (String[] args) { int a = 0x21; int b = 0xff; switch (a) { case 0x21 -> { switch (b) { default -> System.out.println("default"); } } case 0x3b -> System.out.println("3b <- WTH?"); } } } When using the label rules (-> syntax), this is supposed to prevent fallthrough. So in the code, we add a "synthetic" break statement to emulate the breaking in the code (former break; statement which is now implicit). There is a check in the switch statement in org.eclipse.jdt.internal.compiler.ast.SwitchStatement.getFallThroughState(Statement, BlockScope), that tries to identify if a break should be inserted. The check block.doesNotCompleteNormally() returns true in the given example because of the yield statement being the last statement of switch2. So for the case 0x21 in switch1 there is no "synthetic" break statement added and therefore , once the switch2 is generated, it falls through the case 0x3b. I commented out that check and then I got the right behavior because the "synthetic" break statement was added so the break is generated and it no longer falls through the next case. So there are different way to see this: 1) the code should always add the "synthetic" break statement when a block is used with the label rules syntax. 2) make the check to find out if the "synthetic" break statement is needed smarter. In the given example, it should not be skipped. We can discuss that further, but I believe the bug is related to the way we "detect" whether the "synthetic" break statement is required or not.
