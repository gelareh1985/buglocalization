Creation Time 2019-07-09T14:57:35Z
Creating this bug to discuss the larger impact of previewEnabled flag in DOM AST. Bug 548699 added the new API for creating AST with an additional flag to pass the information of preview enablement. As found in https://bugs.eclipse.org/bugs/show_bug.cgi?id=548956#c15 , there are more places where preview flag can have an impact. Initial points to be discussed in this bug: 1. To check the feasibility of using existing deprecated constructor AST(Map options) can be reused so that client can just pass the options and Core can process it. (isPreviewEnabled() API will still be required) 2. Clients to access static properties like org.eclipse.jdt.core.dom.BreakStatement.EXPRESSION_PROPERTY with addition cheach of previewEnabled flag along with AI level check. 3. new propertyDescriptor static method to take previewEnabled or option as input to return appropriate Property descriptor list. 4. Client adds element to the list returned by org.eclipse.jdt.core.dom.SwitchCase.expressions() (or similar lists). But need to be vigilant as only one expression element in the list is alloed without preview Enabled. Core will throw malformed error during conversion if more than one elements found without preview enablement. 5. Method like org.eclipse.jdt.core.dom.BreakStatement.getExpression() can throw error like unsupportedWithoutPreviewError if called for AST without preview enabled.
