Creation Time 2019-03-05T20:46:27Z
The following example was discussed in https://stackoverflow.com/questions/54801795 //--- import java.util.stream.Collectors; import java.util.stream.Stream; public class Test { public static void main(String[] args) { println(Stream.of(42).collect(Collectors.summingDouble(d -> d))); } public static void println(double x) {} public static void println(char[] x) {} public static void println(String x) {} public static void println(Object x) {} } //--- A debugging session shows a missing data flow. Inner inference for summingDouble(..) produces the following capture bound: Collector = capt(Collector) During outer invocation applicability inference for collect(..) we even see this information used for instantiating R#3 to Double. But outer *invocation type inference* for collect(..) cannot see that capture bound created from inner inference. This has the effect that the type Double from the return type Collector is ignored, and a Collector is bogusly inferred instead, leading to all kinds of havoc downstream. Information from inner inference enters the picture via JLS §18.5.2.2. There we find: "C contains all the constraint formulas that would appear in the set C generated by §18.5.2" We already loosely interpret this by copying the Type Bounds created from those constraint, but capture bounds are still missing. (Illegally) including capture bounds in this operation fixes the issue.

Here's in interesting find from comparing the behavior to this example: //--- import java.util.stream.Collectors; import java.util.stream.Stream; public class Test { char[] f; public void test() { f = Stream.of(42).collect(Collectors.summingDouble(d -> d)); } } //--- Here collect is correctly resolved to return Double, despite the target type char[]. The call path that makes a difference is here: ConstraintExpressionFormula.inferPolyInvocationType(InferenceContext18, InvocationSite, TypeBinding, MethodBinding) line: 409 InferenceContext18.computeB3(InvocationSite, TypeBinding, MethodBinding) line: 380 ConstraintExpressionFormula.reduce(InferenceContext18) line: 135 BoundSet.reduceOneConstraint(InferenceContext18, ConstraintFormula) line: 871 InferenceContext18.reduce() line: 1040 InferenceContext18.solve(boolean) line: 991 ParameterizedGenericMethodBinding.computeCompatibleMethod18(MethodBinding, TypeBinding[], Scope, InvocationSite) line: 248 Here we create the capture bound directly into an IC18 which only temporarily has been retargeted for summingDouble() and afterwards will be reset to collect(..). This applies 18.2.1 where indeed data flow is via bound set B3. I think it is fair to consider capture bounds as part of the bound set. The corresponding call path during overload resolution against println(char[]) does not add capture bounds because R theta is already: Collector and hence no wildcard argument is found. I still need to check if this substitution R theta is correct. At that point the ivars in the IC18 are not those from initial constraints, it seems. If experimentally at this point a null substitution is applied, 18.5.2 against target char[] properly fails. This _could_ point towards an alternate fix.

Patch set #2 is an attempt at fixing the issue by a more precise reading of this sentence from 18.5.2: "let θ be the substitution [P1:=α1, ..., Pp:=αp] defined in §18.5.1" referring to "Where P1, ..., Pp (p � 1) are the type parameters of m, let α1, ..., αp be inference variables, and let θ be the substitution [P1:=α1, ..., Pp:=αp]." Yes, substitution is implemented in IC18 by replacing type parameters with matching inference variables, but no, inferences variables created after 18.5.1 should *not* participate in this substitution. This specifically concerns the betas created for capture bounds in 18.5.2. At first sight this distinction may seem irrelevant, because 18.5.2 defines the substitution before creating betas, but inside ConstraintExpressionFormula.reduce(..) inner contexts are intentionally reused if the same expression has to be inferred repeatedly. Perhaps a previous 18.5.2 invocation spills its beta into a subsequent invocation of the same. One more contributing factor: when repeating inference against different overload candidates, then different inference contexts are used, but in our case overloading only affects the outermost method, not those subject to inference, so there's no reason to create new inference contexts.
