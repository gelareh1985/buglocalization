Creation Time 2020-04-14T18:02:26Z
Reference search / call hierarchy has always been somewhat slow for methods with common names / signatures (e.g. "getInstance", "get", "refresh", etc). In our pure Java 8 workspace, it typically takes between 5 to 15 seconds to find references to methods like these; this slow enough to be annoying, but totally manageable. Recently, we've begun to migrate some of the projects in our workspace to Java 11 compliance level and have found that the presence of Java 11 projects dramatically worsens the performance of the "find refs to method with a common name" use-case. Specifically, the performance severely degrades when both of these conditions are met: - You are searching for references to a method that lives in a Java 8 project - One or more large Java 11 projects declare the Java 8 project as a dependency Instead of taking 5 - 15 seconds, finding references to methods like "getInstance" now takes about 40 minutes, which is obviously too slow to be of any use. Flipping all the compliance levels back to 1.8 immediately fixes the problem. The VisualVM sampler reveals that most of the time is being spent in various ZipFile methods (presumably it is reading through the 3rd party jar dependencies of our projects). Based on how much worse the performance becomes, I suspect there is some kind of N-squared issue somewhere causing jar files to be read an excessive number of times.
