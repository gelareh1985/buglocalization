Created attachment 137006 example project I20090522-1710 Constructor completion should boost the relevance of subtypes of the type that is expected by the surrounding context. Example (using SWT): StyledText st= null; st.addSelectionListener(new Selection); In 3.4.2, content assist after 'new Selection' proposed: - SelectionListener (interface) - SelectionAdapter (class) - SelectionEvent (class) - SelectionKey (class) In HEAD, I get: - SelectionListener (anonymous inner type) - SelectionEvent(Event e) (constructor) - SelectionAdapter (anonymous inner type) - SelectionKey (anonymous inner type) The anonymous inner type 'SelectionAdapter' is too far down in the list. In real world examples it gets even worse since there are more constructor proposals. The expected type SelectionListener is known from the context, so all anonymous inner type proposals for subtypes of SelectionListener should get a higher relevance than constructor proposals that only match by name.
