Creation Time 2020-01-24T08:28:53Z
We (on RHEL 7.4 Linux / 4.12) see internally a slight performance drop in JDT compilation time while switching our runtime from 1.8 to 11. I also believe this is due the new "mod-based" packaging and the different implementation in JDT that reads the JRE class files. The compilation time for our use cases (huge projects with lot - 10.000 - 100.000 generated Java files) increases reproducible in the range of 3 to 10%. I think the approach to cache class files from JRE makes total sense and there is no need for Guava to implement some "simple" cache. @Simeon: once we don't have urgent issues anymore, we should try if this caching would have a measurable effect in our case. I can't fur sure guarantee 4.15 goal, but we will investigate that.

My first thought was: shouldn't JavaModelManager cache this? But then I remembered that this manager caches zip files, and so JRT files are probably not covered. I also recall from debug sessions, that reading a JRT image can take long time (in fact, if the debugger had stopped before at certain breakpoints, reading the JRT image could take several minutes in the debugger - possibly a cause of un-JIT'ed code??). OTOH, that field has a TODO to remind us of clearing the cache (e.g., when a JRE is updated on disk?).
