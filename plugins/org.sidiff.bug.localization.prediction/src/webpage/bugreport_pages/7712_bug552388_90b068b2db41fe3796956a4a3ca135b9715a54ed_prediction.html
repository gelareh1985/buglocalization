Follow-up from bug 539105 comment 4: The following code is inconsistently handled by ecj: //--- import java.util.*; class A { class C { T f; } C c = new C(); A(T t) { c = new C(); c.f = t; } void foo(List.C> list) { list.add(new A("").c); list.add(new A(3).c); } T test() { List l = new ArrayList<>(); foo(l); // *1* return l.get(0).f; } void foo2(List> list) { list.add(new A("")); list.add(new A(3)); } T test2() { List> l = new ArrayList<>(); foo2(l); // *2* return l.get(0).c.f; } } //--- At *1* ecj is silent, whereas *2* raises an error: The method foo2(List>) in the type A is not applicable for the arguments (List>) This inconsistency is bogus, because A and A.C have the same parameterization. Since accepting the program would cause CCE at runtime, rejecting the program is mandated. Relevant sections in JLS: ----------------------------------------------------------------- 4.10.2. ... Given a generic type declaration C (n > 0), the direct supertypes of the parameterized type C, where Ti (1 � i � n) is a type, are all of the following: ... C, where Si contains Ti (1 � i � n) (§4.5.1). ... ----------------------------------------------------------------- 4.5.1 A type argument T1 is said to contain another type argument T2, written T2 <= T1, if the set of types denoted by T2 is provably a subset of the set of types denoted by T1 under the reflexive and transitive closure of the following rules (where <: denotes subtyping (§4.10)): ? extends T <= ? extends S if T <: S ? extends T <= ? ? super T <= ? super S if S <: T ? super T <= ? ? super T <= ? extends Object T <= T T <= ? extends T T <= ? super T ----------------------------------------------------------------- It's crucial to see, that type argument containment requires a wildcard on either side and does not recurse into type arguments of parameterized types. This implies the difference between A <: A and List not <: List>
