I'm a bit worried about the practice of disabling tests. Currently in master the JDT/Core test suites have a total of 374 methods called "_test*". While I don't have a very good plan how to prevent this "waste dump" from continually growing, I suggest we should as a first step distinguish tests that are disabled temporarily, just to keep the build going, vs. tests that have been analysed, have their open bug, but are blocked by a well-known factor (spec clarification needed, or technical solution idea lacking, needs a fix in another component etc.). Some JDT/UI tests print to the console a note about being disabled, perhaps with a reference to the bug etc. See org.eclipse.jdt.ui.tests.refactoring.GenericRefactoringTest.printTestDisabledMessage(String) Idea: tests that have an open bug could be re-enabled (technically) but use a specific idiom to print the excuse to the console and return. Then only temp-disabled tests would continue to use the _test* pattern. Or perhaps temp-disabled tests would use a similar early-return pattern, but print to syserr instead of stdout, or use a different string prefix in their message. This would bring a double benefit: we can be sure that long-term disabled tests are linked to a bug, and: we can easily identify all temp-disabled tests. FWIW, in order to avoid dead code warnings, we could say smth like this void testIAmAnalysed() { if (documentedFailure("543210")) return; ... } void testIAmTempDisabled() { if (tempDisabled("sinceBeta15")) return; ... } Both helper methods would print an appropriate method and constantly return true. WDYT? Should I create a new bug for this? PS: Obviously, @Ignore would help but I'm decidedly not proposing to move JDT/Core tests to JUnit 4 at this point :)
