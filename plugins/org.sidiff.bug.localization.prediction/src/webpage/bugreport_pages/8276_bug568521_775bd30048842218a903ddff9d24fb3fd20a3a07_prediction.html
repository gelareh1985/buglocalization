Creation Time 2020-11-04T11:05:52Z
While debugging bug 566262 I've stumbled over this issue. org.eclipse.jdt.internal.core.index.Index.monitor is not volatile but it is accessed from multiple threads, not always inside synchronized blocks => the threads may see a stale, not up-to-date version of the object. And "monitor" reference is used to signal that the index is deleted or not: Example from org.eclipse.jdt.internal.core.search.indexing.AddFolderToIndex.execute(IProgressMonitor) ReadWriteMonitor monitor = index.monitor; if (monitor == null) return true; // index got deleted since acquired Same or similar code exists in other places. I will push a patch.
