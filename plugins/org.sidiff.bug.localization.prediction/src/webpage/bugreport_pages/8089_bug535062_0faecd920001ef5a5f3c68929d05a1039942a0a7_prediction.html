Created attachment 274173 Code sample to show that module annotations are not processed by Eclipse APT Hi, When putting an annotation with @Target MODULE on a module element (module-info.java), it is not processed by annotation processors within Eclipse. In the attachement, I've included 5 module projects: - annotation: which defines two annotations @Bean (target TYPE) and @Module (target MODULE) - beanAnnotationProcessor: which provides an annotation processor for @Bean annotation that print a mandatory warning on the annotated element and message on the standard output - beanAnnotationProcessor: which provides an annotation processor for @Module annotation that print a mandatory warning on the annotated element and message on the standard output - app-bean: which provides a simple class annotated with @Bean and a module declaration annotated with @Module. This project uses the BeanAnnotationProcessor - app-module: which provides a simple class annotated with @Bean and a module declaration annotated with @Module. This project uses the ModuleAnnotationProcessor After installing the annotation, beanAnnotationProcessor and moduleAnnotationProcessor in the maven repository, you can import the projects in Eclipse and activate annotation processing using m2e-apt plugin. As expected, in the app-bean project, the BeanAnnotationProcessor is properly invoked and a warning displayed on the com.example.app.Bean element However, in the app-module project, the ModuleAnnotationProcessor is not invoked while it is properly invoked when building the project with Maven or with javac. I've tested this on latest Eclipse 4.7.3 as well as Eclipse 4.8RC1.

I've been able to debug the issue up to AnnotationDiscoveryVisitor.visit(ModuleDeclaration module, CompilationUnitScope scope) { ... //module.resolveTypeDirectives(scope); // The above call also resolvesAnnotations ... } Although it is true that resolveTypeDirectives call resolvesAnnotations, it doesn't call the visitor's resolvesAnnotations and eventually _annoToElement is not populated with the module's annotation. The following seems to fix the issue: Override public boolean visit(ModuleDeclaration module, CompilationUnitScope scope) { ModuleBinding binding = module.binding; if (binding == null) { return false; } //module.resolveTypeDirectives(scope); // The above call also resolvesAnnotations Annotation[] annotations = module.annotations; if (annotations != null && annotations.length > 0) { ElementImpl element = (ElementImpl) _factory.newElement(binding); AnnotationBinding [] annotationBindings = ASTNode.resolveAnnotations(module.scope, annotations, binding, false); for (AnnotationBinding annotationBinding : annotationBindings) { ReferenceBinding annotationType = annotationBinding.getAnnotationType(); if (annotationBinding != null && Annotation.isAnnotationTargetAllowed(module.scope, annotationType, binding) ) { // binding should be resolved, but in case it's not, ignore it: it could have been wrapped into a container. TypeElement anno = (TypeElement)_factory.newElement(annotationType); _annoToElement.put(anno, element); } } } return true; } This is kind of copy/paste of the resolveAnnotations method of the visitor, the processor is properly called with this change however messages doesn't seem to be displayed in the editor, I can see them in the Error Log but not in the editor when I open module-info.java. Created attachment 274206 Patch I've finally been able to fix it. I've modified the AnnotationDiscoveryVisitor by replacing "module.resolveTypeDirectives(scope);" by a direct call to resolveAnnotations() method in the visitor which properly populates the _annoToElement map. The createProblem() method in BaseMessagerImpl was also missing the MODULE case so it was impossible for the IdeMessagerImpl to display a message on a module compilation unit. I've also added that part to solve the print message issue. Another problem I've discovered while testing the patch is that the getSimpleName() method in ModuleElementImpl was not properly implemented, both getQualifiedName() and getSimpleName() return the same Name object which violates the ModuleElement API when the module's name is composed of multiple identifiers (https://docs.oracle.com/javase/9/docs/api/javax/lang/model/element/ModuleElement.html#getSimpleName--) "If the module name consists of one identifier, then this method returns that identifier. If the module name consists of more than one identifier, then this method returns the rightmost such identifier, which is deemed to be the module's simple name." The fix is also included in the patch.
