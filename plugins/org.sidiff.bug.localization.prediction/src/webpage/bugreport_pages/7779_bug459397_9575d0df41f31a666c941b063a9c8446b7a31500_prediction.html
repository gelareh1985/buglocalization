To explain the issue of "nullBit" vs. "extra": flow analysis keeps a very compact representation of flow information in a group of 4 long fields, where each 4-bit slice through those 4 longs represents one variable, thus we encode status for 64 variables. Once we see more than 64 variables we dynamically allocate more bit vectors, supposedly using the same logic on them. I.e., each bit operation needs to be implemented twice. Apparently, we have an inconsistency between those two implementations in one of the bit-manipulating operations. What puzzles me: we do have a specific test (NullReferenceImplTests, not included in the normal suite, but I run those manually, whenever I change the bit operations). This test should have systematically caught any such inconsistency.
