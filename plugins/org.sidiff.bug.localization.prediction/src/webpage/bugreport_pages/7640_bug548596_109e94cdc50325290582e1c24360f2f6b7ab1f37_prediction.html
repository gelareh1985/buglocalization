The problem is in org.eclipse.jdt.internal.compiler.classfmt.ClassFileReader.getMemberTypes() There is an optimization in the code based on the assumption that if there is an InnerClassInfo for the current class, it appears before the InnerClassInfo of member classes, but for "com/google/actions/api/smarthome/SyncResponse$Payload.class" this assumption is false: innerClassInfos[0] is for com/google/actions/api/smarthome/SyncResponse$Payload$Device innerClassInfos[1] is for com/google/actions/api/smarthome/SyncResponse$Payload I haven't found anything in the jvms that would make this order illegal, so we should remove this optimization.
