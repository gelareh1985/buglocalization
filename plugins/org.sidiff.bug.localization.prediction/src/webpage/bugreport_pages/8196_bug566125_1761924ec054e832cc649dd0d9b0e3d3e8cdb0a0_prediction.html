The following code produces errors: public class Cls { @SuppressWarnings("deprecation") public void bar(int i) { boolean isNumeric = foo( switch(i+1) { case 0 -> new Short((short)0); case 2 -> new Double(2.0d); default -> new Integer((short)6); }); System.out.println(isNumeric); } boolean foo(short data){ return false; } boolean foo(byte data){ return false; } boolean foo(int data){ return false; } boolean foo(float data){ return false; } boolean foo(long data){ return false; } boolean foo(double data){ return true; } } Errors: 1. The method foo(short) in the type Cls is not applicable for the arguments (double) 2. Type mismatch: cannot convert from Double to short I believe this is a problem of SwitchExpression#isBoxingCompatibleWith() calling AllocationExpression#isBoxingCompatibleWith() inadvertently. The overridden method in the latter does a boxing on the target type, which I am not sure we should be doing in case of a yield.

(In reply to Eclipse Genie from comment #2) > New Gerrit change created: > https://git.eclipse.org/r/c/jdt/eclipse.jdt.core/+/167827 The method AllocationExpression#isBoxingCompatible() was introduced by the famous bug 437444. So, far I am unable to construct a test case (other than the one reported now) that hits the RHS of the conditional expression, i.e. the call to isCompatibleWith(). The boxing call, as I said, looks suspicious and safer way would be to just use the other variant of isBoxingCompatible(). Considering that the changes made for the said bug were huge, it's not surprising that this wasn't tested by our tests.
