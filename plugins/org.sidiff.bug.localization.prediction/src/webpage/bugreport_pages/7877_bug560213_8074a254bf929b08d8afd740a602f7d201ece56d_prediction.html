Observations; In the source case, type "@NonNull MyEnum" has this superclass: - Enum In the binary case it's: - Enum<@NonNull MyEnum> The problem detected by NAM.analyse has severity UNCHECKED_TO_UNANNOTATED, so *if* a problem is to be reported, then it should be the new info Unsafe null type conversion (type annotations): The value of type 'List<@NonNull MyEnum>' is made accessible using the less-annotated type 'List' To check whether the problem is justified, let's try to trigger NPE. E.g.: @NonNullByDefault public class EnumProblem { void f(List list) { MyCollections.sort(list); } } class MyCollections { static > void sort(List list) { list.add(null); } } Here we succeed to feed 'null' into the list and if subsequently code in f() dereferences list elements without a null check, then we'll see NPE. Question: do we really need to account for code like "list.add(null);"? In a recent blog post [1] I argued our warning is kind-of good enough to alert about this problem in unannotated code, at list.add(null) we warn: Null type mismatch (type annotations): 'null' is not compatible to the free type variable 'T' Attempts to declare a variable of type T and assign null to it, will always lead to a warning at least. I'm ready to believe that the situation is "safe enough". What's missing in bug 482242 and my blog post: consider type variables as type arguments in the receiving method's signature. Currently, I consider only concrete types and wildcards. Things to consider: 1. don't let TVB.nullBoundCheck() report when we see UNCHECKED_TO_UNANNOTATED. -> simple but possibly incomplete 2. fully integrate TVB as type argument in the solution from bug 482242 3. avoid the difference between source and binary case (regarding annotations within parameterized superclass). [1] https://objectteams.wordpress.com/2020/02/06/interfacing-null-safe-code-with-legacy-code/
