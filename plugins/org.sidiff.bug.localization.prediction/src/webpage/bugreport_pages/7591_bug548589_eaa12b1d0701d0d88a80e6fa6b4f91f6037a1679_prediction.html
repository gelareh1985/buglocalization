(In reply to Eclipse Genie from comment #3) > New Gerrit change created: https://git.eclipse.org/r/144767 OK, every reproducible bug should get at least some initial investigation. Then I found s.t. that looked simple (two almost identical initial constraints generated), but that turned out to be a red herring. Basically every single line of the entire inference engine is a suspect in cases like this, because everything influences everything. In the end I discovered that part of the inference (inner inference for InferenceCheck::compilation_failed) already knew that T#2 = T, but outer inference (for map()) didn't get notice of this, so resolving T#2 to j.l.Object which then failed on the derived constraint T = j.l.Object. On the one hand this seems to be an omission in JLS - each such information flow must be specified, if we want to see a chance that different compilers accept the same language. But in previous communication with Oracle I learned that javac is doing a lot of passing information from one level to the other which is not visible in JLS. So apparently, more is better, even when it brings us further away from JLS. :-/ Let's see if it breaks anything in a full test run.
