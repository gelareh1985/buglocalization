Using plexus-compiler-eclipse 2.8.6 to run ecj, we now get messages emitted by APT processors with Diagnostic.Kind.NOTE reported as warnings. Which breaks CI builds that implement a zero-warnings policy. The cause is org.eclipse.jdt.internal.compiler.apt.dispatch.BaseMessagerImpl.createProblem(), which maps any non-error Diagnostic.Kind to ProblemSeverities.Warning. Line 186ff: int severity; switch(kind) { case ERROR : severity = ProblemSeverities.Error; break; default : // There is no "INFO" equivalent in JDT severity = ProblemSeverities.Warning; break; } The comment appears to be outdated. There is a ProblemSeverities.Info, and it seems to be handled in the BatchCompiler's Main and also in the JSR-199 EclipseCompilerImpl. I suggest to map Diagnostic.Kind.NOTE and Diagnostic.Kind.OTHER to ProblemSeverities.Info. For instance, the Hibernate JPA metamodel generator uses Diagnostic.Kind.NOTE to print its version, and uses Diagnostic.Kind.OTHER for debug messages. Or map Diagnostic.Kind.NOTE to ProblemSeverities.Info, and Diagnostic.Kind.OTHER to zero (no bits set). That would be consistent with the implementation in EclipseCompilerImpl, but I have no idea what effect a severity zero might have elsewhere.
