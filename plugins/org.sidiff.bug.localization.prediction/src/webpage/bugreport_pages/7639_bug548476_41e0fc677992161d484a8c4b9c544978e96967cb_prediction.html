The grammar changes for switch expressions have broken code completion also for old Java versions (see e.g.: bug 547256). OTOH, there's no use in fixing this, because the grammar for switch expressions changes for Java 13. Still Eclipse 4.13 cannot ship that new grammar. Hence in bug 542560 agreement was reached to pull the plug on the out-dated preview feature to get completion back on track.

(In reply to Eclipse Genie from comment #16) > New Gerrit change created: https://git.eclipse.org/r/144829 An alternate approach: This is an attempt to touch the middle ground for preview: Assumptions: 1. Those who use the code without preview should see things as it was � no regressions. 2. Those who use preview can live with some reduced functionality wrt completion. We cannot support two different grammars for this, however, we can effectively direct the parser to take two different routes, effectively supporting two slightly different grammars. For that, the following synthetic token BeginPreviewMarker is introduced with the following grammar rules: BreakStatement ::= 'break' ';' /.$putCase consumeStatementBreak() ; $break ./ BreakStatement ::= 'break' Identifier ';' /.$putCase consumeStatementBreakWithLabel() ; $break ./ /:$readableName BreakStatement:/ BreakStatement ::= 'break' BreakPreviewMarker ';' /.$putCase consumeStatementBreak() ; $break ./ BreakStatement ::= 'break' BreakPreviewMarker Expression ';' /.$putCase consumeStatementBreakWithExpressionOrLabel() ; $break ./ /:$readableName BreakStatement:/ First two is a revert to the previous (which again is reinstated for Java 13 � so we are good there) and the last two are routing agents. This routing will be done via introduction of synthetic tokens at scanning time. Advantage I see that there is no rollout of the Java 12 work needed here to support the previously existing functionality. @Stephan, @Dani, all: comments/suggestions welcome.
