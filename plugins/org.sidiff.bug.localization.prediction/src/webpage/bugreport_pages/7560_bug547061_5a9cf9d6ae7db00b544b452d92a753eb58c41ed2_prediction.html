(In reply to Stephan Herrmann from comment #3) > The example is rejected since the fix for bug 521159. > > Simply commenting this new line in MessageSend.resolveType() flips the > behavior: > > > scope.referenceContext().tagAsHavingIgnoredMandatoryErrors(IProblem. > UndefinedMethod); > > I don't yet have a theory how that line could cause the effect we are > observing here. The connection is roughly like this: on first attempt to resolve the lambda we find a MessageSend that is unresolvable due to non-proper target type, so we tagAsHavingIgnoredMandatoryErrors which is propagated into the enclosing lambda(s). This flag then prevents calls to copy.analyzeExceptions() so LE#thrownExceptions is never initialized. This information is later missing when trying to resolve a ConstraintExceptionFormula. Since the change from bug 521159 didn't fix any user-observable problem, I'm simply reverting it here (while retaining the test from that bug of course): (In reply to Eclipse Genie from comment #5) > New Gerrit change created: https://git.eclipse.org/r/141893
