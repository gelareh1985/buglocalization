This was reported to us here: https://github.com/spring-projects/sts4/issues/535 Summary, it seems the Default / Built-in Eclipse formatter now has the handling of formatter on/off tags disabled so it no longer respects them out of the box. I am unsure since when this is, or if it was done deliberately or by accident. I have reproduced the problem with Eclipse 4.16 for Java Developers (the precise download used was: /home/kdvolder/Downloads/eclipse-java-2020-06-R-linux-gtk-x86_64.tar.gz). I have also reproduced it with current/latest STS RC build which is based on latest Eclipse 4.17 RC. So I gather it is a problem probably since Eclipse 4.16 and still an issue in 4.17. Here is an example that you can use to try it for yourself: package testing; public class Fronesna { public static void main(String[] args) { // @formatter:off SpringApplication.run( DemoApplication.class, args); // @formatter:on } } To reproduce: - download fresh copy of Eclipse 4.16 - create new workspace - Create a new plain java project. - Put the sample code in it (ignore compilation errors, they don't matter). - Select all the text in the editor. - Format it with 'Source >> Format' from context menu. => You will see that the formatter tags are not obeyed and the text between the tags is formatted. You can also check that the formatter tags preference is disabled. This I supppose is why the tags are disobeyed. If we create a custom formatter and change this setting then it works as expected.

Creation Time 2020-09-25T14:27:36Z
Okay, thanks for verifying this and so I beleave you that this was of for a long time. So indeed you can classify this bug as 'invalid' on those grounds. However, might we instead repurpose this bug to ask the question whether perhaps it might be a good idea to change this default? I.e. consider this a request for a small enhancement instead? Personally I really do not care about that preference as I don't really use it. But some of our users do however. I don't see any advantage at all to having this feature turned of by default, is there any ill effect we anticipate from having it turned on? On the other hand, to those users who do like to use the feature, enabling it manually is a bit of PITA (and that is assuming the can figure out they can enable it and how which is not a given since the Eclipse preferences are a real challenge to navigate). So, in short, I think enabling this by default would hurt no-one and would be helpful to some. So probably it is a good idea, regardless of the history.
