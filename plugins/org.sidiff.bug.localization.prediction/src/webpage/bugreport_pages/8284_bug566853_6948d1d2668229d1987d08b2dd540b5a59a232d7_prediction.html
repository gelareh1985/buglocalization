@Kalyan: as for your offline query on �how do we know whether the hashCode, toString and equals of a record is synthetic or not� � here is a issue with the current checks and a possible solution: Issue: First, from a dom MethodBinding perspective, the isSynthetic() uses the AccSynthetic flag to check for a method being generated. This will not work here since AccSynthetic is not set for these generated methods � why? Because see Sec 4.7.8 of record companion doc for JVM changes (latest as of now available at [http://cr.openjdk.java.net/~gbierman/jep395/jep395-20201019/specs/records-jvms.html#jvms-4.7.8] � to quote: � A class member that does not appear in the source code must be marked using a Synthetic attribute, or else it must have its ACC_SYNTHETIC flag set. The only exceptions to this requirement are compiler-generated members which are not considered implementation artifacts, namely �. the implicitly-declared members of enum and record classes (JLS 8.9.3, 8.10.3).� Now if you refer to the Sec 8.10.3 of changes in JLS for records [latest as of now available at http://cr.openjdk.java.net/~gbierman/jep395/jep395-20201019/specs/records-jls.html#jls-8.10.3], you can see these methods are referred to there- ie equals(), hashCode() and toString(). Hence it will not comply to spec if set AccSynthetic and consequently setting AccSynthetic flag for these members is not an option. Possible solution: So the alternate would be for the dom ast MethodBinding to define a method and that simply checks whether the given method binding specifically of hashCode, toString or equals() of a record is an instanceof SyntheticMethodBinding (compiler). SMB is created only if the user has not provided the method.I prefer this method is provided as an @noreference method (just shy of making it an API since I don�t see any reason to do that except for the consumption of ui).
