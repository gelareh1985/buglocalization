I20150324-0800 (Found through the fix for bug 463252.) When resource leak problems are enabled in a 1.7 project, I get a potential problem on the first "zipFile" variable, and a definite problem on the second "zipFile" variable here (good): /** * Inline zipFile => potential resource leak disappears. */ public class Try17 { void potential() throws IOException { ZipFile zipFile= getZipFile(); String name= zipFile.getName(); System.out.println(name); } void definite() throws IOException { ZipFile zipFile= new ZipFile("bla.jar"); String name= zipFile.getName(); System.out.println(name); } ZipFile getZipFile() throws IOException { return new ZipFile("bla.jar"); } } When I inline "zipFile", the definite resource leak stays, but the potential leak disappears. I don't see why the absence of a local variable should make a difference for a potential leak. The Javadocs of JavaCore.COMPILER_PB_UNCLOSED_CLOSEABLE and COMPILER_PB_POTENTIALLY_UNCLOSED_CLOSEABLE also don't reveal why that would be intended. Both only talk about a "local variable", although the definite problem also appears on an expression.
Creation Time 2015-03-27T23:21:55Z
I guess this is just a bug in how we detect expressions possibly feeding a fresh resource into the flow: String name = new ZipFile("bla.jar").getName(); -> an allocation expression is a known source of a fresh resource ZipFile zipFile = getZipFile(); -> each local variable initialization / assignment is checked for resource types String name = getZipFile().getName(); -> we fail to check the message send that's not a RHS in a local variable initialization / assignment. Interestingly, we wouldn't even need to do any flow analysis in this particular case: the resource is not visibly flowing anywhere; it's always a potential leak. [resource] bugs are on my agenda for M7 anyway :) BTW: I see the javadoc very well in line with the current behaviour regarding potential(): after inlining there is no local variable holding (and possibly leaking) a resource. It's only the warning against "new ZipeFile(..).getName()" that's beyond the documented behavior. But we seem to agree that we should report more warnings (after inlining in potential()) rather the fewer (after inlining in definite()).

Creation Time 2015-03-31T17:32:10Z
(In reply to Markus Keller from comment #3) > When this bug gets fixed, the Javadocs should be changed to > "[..] if a local variable or expression holds a value of [..]". Perhaps that's indeed what the javadoc should say. But to be explicit among us, here's the previous strategy and my planned changes to it (we'll probably never handle all kinds of expressions, see below): So far resources can come into the picture - via local variables (declaration with init or assignment) - allocation expressions, immediate or wrapped in: - cast / conditional / assignment expression Planned addition: - message send returning a (auto)closeable type (problem will be "potential" at best, the target method may have obtained the resource from some shared instance which may retain the responsibility to close). Deliberately not considered are: - field references - fields are not owned by any particular method, impossible to tell which method is *supposed* to close the resource - method arguments - unclear ownership, caller is likely to continue working with the resource - array references - we don't really have a chance to do meaningful flow analysis for array elements. - I hope this is a rare-enough use case to be left unhandled. Let me know if you see the need to add more expression kinds beyond message sends.
