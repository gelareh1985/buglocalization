Creation Time 2019-08-24T17:56:05Z
Have a minimal modular project (with no dependencies) with a main class test.Test, and verify that the class is runnable using Run as > Java Application. Next, move module-info.java from the source folder to the project's root folder. Now Run as > Java Application complains (editor): "Editor does not contain a main type" or (Package Explorer): "Selection does not contain a main type" Debugging revealed that the project still answers non-null to IJavaProject.getModuleDescription(), which causes MatchLocator.skipMatch(JavaProject, PossibleMatch) to answer 'true' during search of a main method. Next I played with renaming module-info.java to no-module-info.java and back. Now launching succeeds but the application throws: Error occurred during initialization of boot layer java.lang.module.FindException: Module MyModule not found In that state, search was OK, but launch invented a CLI argument -m MyModule/test.Test although no module MyModule exists.

(In reply to Eclipse Genie from comment #1) > New Gerrit change created: https://git.eclipse.org/r/150646 Meanwhile I observed another symptom of the same root cause: given a Java class that accesses javax.xml.bind.annotation.XmlRootElement via "requires java.xml.bind;", removing module-info.java should result in compile errors, because java.xml.bind is not a default module when compiling for the unnamed module. Due to the same caching, the compile error occurred only after closing/re-opening the project. The test case tries to simulate this situation, but failed to capture the failure without the fix. Perhaps some listener is needed to reproduce the problem. Theory: perhaps such listener recreates the SourceModule into the JavaProjectElementInfo right after it has been initially removed? I might consider writing a test in JDT/UI where we can simulate the same D&D gesture that was used in my manual testing. Another issue with the test: it is quite narrow, as it needs a JRE of version 9 or 10, but not larger. I haven't yet found a better example using only modules that exist in all recent versions of JRE. The fix just adds more calls to projectCachesToReset.add(..), as are done in the vicinity already. I can confirm that this change fixes the real problems observed in the IDE, those from comment 0 as well as the new symptom.
