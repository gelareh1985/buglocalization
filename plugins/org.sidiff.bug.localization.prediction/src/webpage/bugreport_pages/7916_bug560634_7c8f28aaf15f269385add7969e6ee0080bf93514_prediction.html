Creation Time 2020-02-28T11:07:42Z
When a Java project uses project specific settings for "Java Compiler" and the checkbox "Use '--release' option" is enabled, the following message appears in the eclipse log when any file in the project is changed after starting Eclipse: JavaBuilder: Starting build of RebuildTest @ Fri Feb 28 10:24:12 CET 2020 About to read state RebuildTest Successfully read state for RebuildTest JavaBuilder: New location: Classpath jrt file C:\Program Files\Java\jdk-11.0.5+10\lib\jrt-fs.jar with --release option 11 != old location: Classpath jrt file C:\Program Files\Java\jdk-11.0.5+10\lib\jrt-fs.jar JavaBuilder: New locations: /RebuildTest/bin C:\Program Files\Java\jdk-11.0.5+10\lib\jrt-fs.jar JavaBuilder: Old locations: /RebuildTest/bin C:\Program Files\Java\jdk-11.0.5+10\lib\jrt-fs.jar JavaBuilder: Performing full build since classpath has changed JavaBuilder: Clearing last state : State for RebuildTest (#0 @ Fri Feb 28 10:23:33 CET 2020) FULL build [...] So, a full rebuild is triggered. Further changes do not necessarily cause a rebuild. I first noticed this behaviour in Eclipse 2019-12. Strangely, it does not happen when "Use '--release' option" is enabled in the compiler settings of the workspace and the project doesn't use project-specific settings. To produce the log, I used a .options file with the lines org.eclipse.jdt.core/debug=true org.eclipse.jdt.core/debug/builder=true and I started eclipse.exe with parameters -debug -consolelog

Thanks Jonas for a clear and helpful report (In reply to Jonas Thedering from comment #0) > JavaBuilder: New location: Classpath jrt file C:\Program > Files\Java\jdk-11.0.5+10\lib\jrt-fs.jar with --release option 11 > != old location: Classpath jrt file C:\Program > Files\Java\jdk-11.0.5+10\lib\jrt-fs.jar So we have one ClasspathJrtWithReleaseOption and one regular ClasspathJrt. As per ClasspathJrtWithReleaseOption.equals() they are not equal. It seems the root cause is in org.eclipse.jdt.internal.core.builder.State.write(DataOutputStream), where checks are done in wrong order: } else if (c instanceof ClasspathJrt) { ClasspathJrt jrt = (ClasspathJrt) c; ... } else { ClasspathJrtWithReleaseOption jrt = (ClasspathJrtWithReleaseOption) c; ... the else branch is dead code because ClasspathJrtWithReleaseOption extends ClasspathJrt. @Jay, I guess all of us had hoped to never revisit bug 540922, but that's where this bug got introduced, and I have no clue what dragons are lurking once we stir them ... =:-0
