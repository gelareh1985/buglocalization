This is arguably a bug rather than a mere "enhancement request" - Eclipse shows both warnings (X collides with a package) and errors (X collides with a type) where no mistake exists (the code builds without errors or warnings). There is a partial workaround which helps mitigate the problem: In the "Problems" view, open the view's menu and select "Filters..."; that'll open the "Filters" dialog. In there, use the "Description" section to set Text: "doesn't contain" with the field "collides with a". This doesn't prevent Eclipse mistakenly highlighting these as warnings/errors within the source code editors or the package explorer, but it does at least trim down the noise within the "Problems" view itself.

(In reply to Anton Tanasenko from comment #16) > (In reply to Stephan Herrmann from comment #11) > > tentatively putting it on our agenda, but this remains as "helpwanted". > > Is there any chance it could go into march release? Making this error configurable is a laudable attempt, but since the error is actually wrong, I'm now trying to fix the root of it. Unfortunately, semantics are still blurry, given: a/B.java a/B/c/D.java We have packages a and a.B.c, but does package a.B exist and create a conflict? Per explanation in JLS ยง7.1 we have indeed a conflict: "If there is a package named mouse and a member type Button in that package (which then might be referred to as mouse.Button), then there cannot be any package with the fully qualified name mouse.Button or mouse.Button.Click" Just a little further down, it says, however, that the package hierarchy has no semantic significance: "The hierarchical naming structure for packages is intended to be convenient for organizing related packages in a conventional manner, but has no significance in itself other than the prohibition against a package having a subpackage with the same simple name as a top level type (ยง7.6) declared in that package" And in other places it asserts that for a package to exist it must be declared in a compilation unit, which is *not* given in the example. Anyway both compilers complain, so we should leave this as it is. To fix this issue, we'll need to closer inspect directory a/B: does it imply a package or is it just a directory? If it contains - a .class file, then it's a package - a .java file - *if* the file contains any declarations then it's a package - a sub directory, then the analysis may need to be applied recursively. The last part looks a bit ugly performancewise. The same for the requirement to read and parse a .java file to see if it declares a package, which I'll skip for now.
