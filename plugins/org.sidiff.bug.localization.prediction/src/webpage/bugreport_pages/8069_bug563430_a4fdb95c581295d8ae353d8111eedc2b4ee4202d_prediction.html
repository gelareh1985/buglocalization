(In reply to Eclipse Genie from comment #1) > New Gerrit change created: https://git.eclipse.org/r/163361 A few words about this design: First of all, I've used build_parser script of bug 562044 fame for generating rsc++ files. And there is a change in GenerateParserScript file as well. Initial thought was we might have to change jikespg itself but that was found unnecessary (probably except for a byte array overflow - more on this later). Direct usage of jikespg without any modification gets us Tokennamenon-sealed which is not a valid variable name in TerminalTokens - However, since this is a file which is generated already, we intervene here with a simple fix in GPS to replace with a spooky variant - TokenNamenonHyPhEnAteDsealed (please feel free to suggest!) All's well until we hit the problem in Scanner - "non-sealed" would end up as three tokens normally, - TokennameIdentifier, TokennameMinus,and TokennmeIdentifier - this issue is taken care by the change in the internalScanIdentifierOrKeyword(). Every other change in the git (atleast the first version - at the time of writing this) is typical. Coming back to jikespg, we get the following warnings: ***Warning: Base Check vector contains value > 127. 16-bit words used. ***Warning: Terminal symbol > 127. 16-bit words used. non-sealed may be an invalid variable name. Atleast the first two need to be investigated (I assume the last one is taken care). Again, at the point of writing this, the patch is WIP - please provide your early inputs especially for the changes listed above.

I agree to doing magic in Scanner. What I don't understand: if Scanner synthesizes a token that would not normally be scanned as one token, why worry about the spelling 'non-sealed' in the grammar? Where, aside from internalScanIdentifierOrKeyword() do we need to know the source representation of that keyword? See all the tokens which we synthesize with no correspondence in source code at all, like all those TokenNameBegin*. BTW, why not simply say TokenNamenon_sealed? The jikespg messages could be a signal that we should check if some of the arrays must be upgraded. E.g.: in Object Teams I already had to change Parser#rhs from byte[] to char[], and build the file using buildFileOfIntFor() instead of buildFileOfByteFor() - then initialize at runtime using readTable() instead of readByteTable(). I also had to add a cast inside method tAction(). See https://git.eclipse.org/c/objectteams/org.eclipse.objectteams.git/tree/org.eclipse.jdt.core/compiler/org/eclipse/jdt/internal/compiler/parser/Parser.java and search for marker "//{ObjectTeams" I don't know which of these might become necessary in JDT now, but I can confirm, these changes "don't hurt" :) While I'm at it: In Object Teams I also made some tweaks in Scanner, where JDT has declarations like: public /*static*/ final char[][][][] charArray_length The history doesn't give a hint on why /*static*/ rather than static. OTOH, I don't recall seeing any problems resulting from non-static, it just looked wrong to me.
