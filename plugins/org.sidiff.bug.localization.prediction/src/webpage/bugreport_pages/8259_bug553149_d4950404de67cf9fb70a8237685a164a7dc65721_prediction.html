There are several areas that need attention wrt to code selection involving a pattern matching instancof. For example, the following code: public class X { protected Object x = "FIELD X"; @SuppressWarnings("preview") public void f(Object obj, boolean b) { if ((x instanceof String x) && x.length() > 0) { System.out.println(x.toLowerCase()); } } } The first "x" in the pattern matching refers to the field "x". But code selection throws up an error citing the current selection can't be opened in the editor. This, I believe is due to the missing handling in SelectionParser#consumeInstanceOfExpression() for pattern variable. The other big problem, all other references of "x" take the cursor to the field declaration rather than the pattern variable. These are must-fix IMO.
Creation Time 2020-09-11T08:36:42Z
It's going to be extremely complicated to make things work for selection when the pattern variable shadows a field. Even though resolution fully happens without the need for flow analysis (with the help of a pre analysis (bug 562824)), the fact that code assist only creates minimal nodes, which means we might have lost some details. For things to work, we should fully resolve all the statements involved. For e.g., when an IF statement's condition contains a pattern variable and throws an exception in the THEN statement, we must resolve the complete IF/THEN statement. I wonder whether this is worth it. My plan is to fix the basic cases (not involving a field shadowing) and take up the rest later.
