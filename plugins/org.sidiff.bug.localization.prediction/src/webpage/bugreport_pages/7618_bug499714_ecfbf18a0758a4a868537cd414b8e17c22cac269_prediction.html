Observations at different versions (always with -err:+nullAnnot -warn:+null,+inheritNullAnnot): 4.4.2 - 4.5.2: t.get().toString(); ^^^^^^^ Potential null pointer access: The method get() may return null 4.6 - 4.12: t.get().toString(); ^^^^^^^ Potential null pointer access: this expression has type 'T', a free type variable that may represent a '@Nullable' type I could reproduce the 'conflict' error message only by changing the scope of @NonNullByDefault to affect the entire source file, then: @Override public String get() { ^^^^^^ The default '@NonNull' conflicts with the inherited '@Nullable' annotation in the overridden method from Type This looks correct to me. OTOH, @NonNullByDefault is *not* applied to the return type of Type.get, because of the rule "Wildcards and the use of type variables are always excluded from @NonNullByDefault." One problem I see is in the wording re 'free type variable', where in fact is constraint to be @Nullable. This latter problem I'll try to fix and then all should work as designed.
