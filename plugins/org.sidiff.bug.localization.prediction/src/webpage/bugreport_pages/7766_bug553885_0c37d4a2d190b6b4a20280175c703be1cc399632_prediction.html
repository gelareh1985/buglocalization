The patch resolves the issue, but I see a danger of confusing two types of "bridges": - The JDK bugs speak of lambda bridges being generated as a workaround to a JVM issue. We call these "implicit lambdas", selector is s.t. like lambda$0. - RE.shouldGenerateImplicitLambda() avoids generating an implicit lambda, if we can assume that a "bridge method" will be generated into the accessible receiver class, which makes the target method available to the public. Selector is the same as the original target method. The problem in this bug only occurs if *neither* of these is generated. Additionally, after compiling the given test case with Andrey's patch I see the following invocation inside the implicit lambda: 1: invokevirtual #44 // Method p1/BImpl.amount:()I This happens, because the Optional is inferred as Optional. For a moment I was tempted to change code gen such as to use BImpl for the bootstrap method. To fully elaborate our options I am adding two more tests, so we have: (a) the original test (b) a variant where type inference is led into using Optional rather than Optional. (c) the same structure but using classes instead of interfaces. Without the patch Bootstrap is generated using #70 REF_invokeInterface p1/B.amount:()I (a) generates no implicit lambda, no bridge method => fails (b) generates no implicit lambda, no bridge method => fails (c) generates no implicit lambda, but a bridge method => succeeds With my alternate attempt (different bootstrap generated): (a) no implicit lambda, but bootstrap using #66 REF_invokeVirtual p1/BImpl.amount:()I no bridge method => succeeds (b) no implicit lambda, but bootstrap using #70 REF_invokeInterface p1/B.amount:()I no bridge method => fails (c) same as above => succeeds With Andrey's patch: Bootstrap is generated using #70 REF_invokeInterface p1/B.amount:()I (a) implicit lambda containing 1: invokevirtual #52 // Method p1/BImpl.amount:()I => succeeds (b) implicit lambda containing 1: invokeinterface #54, 1 // InterfaceMethod p1/B.amount:()I => succeeds (c) same as above => succeeds Finally, one might be tempted to change generation of regular bridge methods for the default method. This, too, would solve the current issue, by essentially reverting bug 404690 (prior to which we generated regular bridge methods to word around another JVM bug). Between the two working solutions (with implicit lambda vs. with regular bridge), Andrey's patch is to be preferred, because it generates code similar to javac, in areas where apparently following the specs is not sufficient.
