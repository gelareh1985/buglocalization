Thank you Jay, your hint that the package name is invalid was exactly right. While distilling down the test case to remove javapoet I logged the name, and found that it was different in javac vs Eclipse! It comes down to the processor misusing the javax.lang.model: calling Package.toString() returns the qualified name in javac, but returns 'package qualifiedName' in Eclipse! So in Eclipse we attempt to place our generated sources in package 'package com.somewhere.generated' which of course is not valid. The fix is easy: change `getPackage().toString()` to `getPackage().getQualifiedName().toString()` in your annotation processor and the issue vanishes. This is therefore not an Eclipse bug, but rather an implementation difference. (it does not appear that toString is specified particularly in this case) Eclipse could choose to more thoroughly validate processing.Filer method parameters: "fail fast" when the class name includes invalid characters including space probably would have saved both myself and the original reporter hours of time. But I appreciate that at the end of the day it is our responsibility to hold the tool right :)
