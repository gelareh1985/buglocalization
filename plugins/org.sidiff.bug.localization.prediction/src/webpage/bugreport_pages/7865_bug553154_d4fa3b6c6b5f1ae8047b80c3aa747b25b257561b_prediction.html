Since you accepted my point about the grammar of FieldDeclaration not fitting record component in https://git.eclipse.org/r/156247, there's a similar issue with using MethodDeclaration for compact constructor. MethodDeclaration javadoc states that there are always parenthesis after the method name, even if the parameters list is empty. This will no longer be the case with compact constructors. Is this change intentional? There already are two distinct grammars described in the javadoc (for normal method and a constructor), so adding a third one doesn't seem unacceptable, but still a bit problematic. An API client like the formatter, if left unapdated, will throw an exception because of wrong assumptions. AFAIK earlier changes have never caused this kind of issue, the client would just skip new types of nodes it's not aware of. I don't see a class that can be used instead, Initializer seems to be the closest, except for missing name.

We had an internal discussion now, and we are in favour of adding a new Grammar rule for Compact Constructor in Javadoc. Reasons behind this discussion: 1. It is a part of preview feature and it can change so if required we will have a relook at this once it becomes a standard feature. 2. JDT UI will be impacted a lot if we add a new node as CompactConstructor node. 3. In DOM, we will have to add a new AbstractMethodDeclaration node and make CompactConstructor as a sibling of Method Declaration. Unless we have some challenges in "Formatter", which is difficult to handle without a new node, we will like to go ahead with this approach.
