Observed after the fix for bug 562727: locally running RunJDTCoreTests may trigger this exception: java.nio.file.ClosedFileSystemException at jdk.zipfs/jdk.nio.zipfs.ZipFileSystem.ensureOpen(ZipFileSystem.java:1105) at jdk.zipfs/jdk.nio.zipfs.ZipFileSystem.checkAccess(ZipFileSystem.java:332) at jdk.zipfs/jdk.nio.zipfs.ZipPath.checkAccess(ZipPath.java:830) at jdk.zipfs/jdk.nio.zipfs.ZipFileSystemProvider.checkAccess(ZipFileSystemProvider.java:185) at java.base/java.nio.file.Files.exists(Files.java:2439) at org.eclipse.jdt.internal.core.builder.ClasspathJrtWithReleaseOption.initialize(ClasspathJrtWithReleaseOption.java:111) at org.eclipse.jdt.internal.core.builder.ClasspathJrtWithReleaseOption.(ClasspathJrtWithReleaseOption.java:77) at org.eclipse.jdt.internal.core.builder.ClasspathLocation.forJrtSystem(ClasspathLocation.java:147) at org.eclipse.jdt.internal.core.search.matching.JavaSearchNameEnvironment.mapToClassPathLocation(JavaSearchNameEnvironment.java:164) at org.eclipse.jdt.internal.core.search.matching.JavaSearchNameEnvironment.computeClasspathLocations(JavaSearchNameEnvironment.java:133) at org.eclipse.jdt.internal.core.search.matching.JavaSearchNameEnvironment.(JavaSearchNameEnvironment.java:77) I succeeded to reduce the scenario to running just these three tests in order: org.eclipse.jdt.core.tests.builder.Bug549646Test.testCompilerRegression() org.eclipse.jdt.core.tests.compiler.regression.ModuleCompilationTests.testReleaseOption10() org.eclipse.jdt.core.tests.model.ModuleBuilderTests.testConvertToModule() The first test creates ClasspathJrtWithReleaseOption which initializes a ZipFileSystem from which various things are read, including the new CtSym. The second test uses org.eclipse.jdt.internal.compiler.batch.FileSystem with a child org.eclipse.jdt.internal.compiler.batch.ClasspathJep247. The latter initializes its field #fs from FileSystems.getFileSystem(uri), which may return the cached instance ZipFileSystem from the first test. At the end, this test invokes ClasspathJep247.reset(), which closes the file system. The third test, is similar to the first, but now we will use the cached CtSym, from which we obtain the existing ZipFileSystem, that had been initialized by the first test, but closed by the second test. Ergo: by leaking a CtSym, we may later access a ZipFileSystem that has already been closed, and thus is unusable. Note, that use of FileSystems.getFileSystem(uri) is generally protected to handle the case when a file system has been closed, and a newFileSystem() will be requested. Caching CtSym, however, bypasses this policy.

(In reply to Eclipse Genie from comment #1) > New Gerrit change created: https://git.eclipse.org/r/163463 This contains a special purpose test suite (not integrated into regular builds, just for manual invocation). Without the fix this on my machine deterministically triggers the bug when run on JDK 11 with -Dcompliance=11. I was tempted to let ClasspathJrtWithReleaseOption.cleanup() unregister the CtSym instance, but probably this would defeat part of the savings in bug 562727. Turned out, it's enough to check getFileSystem().isOpen() upon access. In fact, with all this JVM-wide sharing, this looks quite fragile. Any plugin could request the FileSystem for the same URI, close it and -- boom -- JDT be broken. Another thing I saw: ZipFileSystem can also be closed from it's own finalize(), so caching this guy (directly or indirectly via CtSym, e.g.,) will also prevent finalization and thus hold on to any resources belonging to the ZipFileSystem. Is this a problem? I'm not sure if classes from the batch compiler are truly relevant in IDE settings, but unless s.o. has a solid proof, I'd rather assume that file system closing *can* happen, rather than be sorry. For that reason I propose this for RC1. Comments?
