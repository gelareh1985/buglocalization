What I've also saw is that in many places in JDT core/UI where AST.JLS12 or AST.JLS12_INTERNAL were used, changing the code from if(this.ast.apiLevel >= AST.JLS12_INTERNAL) to if(this.ast.apiLevel >= AST.JLS12_INTERNAL && this.ast.isPreviewEnabled()) fixed *most* of the tests (with additional NPE fixes in UI code, with similar checks). The change I took the idea from was for example for ASTRewriteAnalyzer in commit 357185ef691b346646bc98aa38ba4950222dbc6a, so I've wondered that we still have many unchanged "if" in both UI / core code and tried to play with that. From now on there will be a requirement of new static method propertyDescriptors(int, boolean) with previewEnabled as input for all the  DOM AST elements which have new properties as a part of preview feature. Like SwitchCase and BreakStatement will have this new API for Java 12 and SwitchCase and Yield will have for Java 13. This also means a lot of work and change in ensuring that methods like org.eclipse.jdt.core.dom.SwitchCase.expressions() is not used by clients to add more than one element when preview is enabled.
