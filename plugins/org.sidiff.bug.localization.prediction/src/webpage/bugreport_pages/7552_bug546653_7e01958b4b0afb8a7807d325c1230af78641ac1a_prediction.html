Creation Time 2019-04-23T09:46:38Z
In order to implement more comprehensive control over JEP 261 related options, I'm proposing two changes in IClasspathAttribute: To add constant ADD_OPENS, which will allow to define the equivalent of CLI --add-opens using a classpath attribute. This particular attribute is not relevant for compiling, but should be passed on into launching. To enhance the existing attribute with constant PATCH_MODULE: currently, this attribute has a single value: the name of the module being patched, but per JEP 261 the correpsonding option has this structure: --patch-module =()* I propose to optionally support the "=..." part, so that different sources can patch different modules during the same build. In our case could be the path to either a project or a source folder. I suggest to use workspace paths. Referring to a source folder should have the semantics of associating all contained compilation units to the specified module, which must be served by the classpath entry, to which the attribute is attached. Referring to a project should have the same semantics as enumerating all source folders of that project. Compatibility with current semantics of that attribute can be established simply, if we interpret absence of the "=..." part as referring to the current project (the enclosing project of this .classpath), and thus referring to all source folders of the current project. The tooling should enforce the following consistency constraints: - for each given module, there can be at most one patch-module attribute (which can, however, list an arbitrary number of sources). - for each source folder can only patch one module (at most), i.e., the same source folder cannot be referenced from different patch-module attributes, nor is it legal to mention a source folder plus its containing project in different patch-module attributes.

I realized we need to pay more attention to multi-valued attributes. So I updated the javadoc like this: - add-opens & add-exports: mention packing as : separated lists - patch-module: here a single : would conflict with the use of File.pathSeparator within a single definition (associating several locations with one module).
