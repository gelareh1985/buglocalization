Both javac & ecj are consistent in their respective behavior in all versions since Java 8: javac: accept ecj: reject When debugging ecj, this is what I see regarding the final method invocation: receiver type is capture#2-of ? extends capture#1-of ? which has this superinterface: DslStep2 where we find this method: public abstract capture#3-of ? extends capture#2-of ? extends capture#1-of ? doSomething() superclass of the initial capture is capture#1-of ? again we look at the superinterface: DslStep2 where we find this method: public abstract capture#4-of ? extends capture#1-of ? doSomething() since both methods have the same argument list (), overloading cannot be resolved. I doubt that JLS is sufficiently precise to tell us how to handle this case, but clearly reporting ambiguity is not expected.
Creation Time 2019-05-31T19:12:59Z
Interestingly on behalf of bug 434044 comment 20 we already added a non-JLS check in this exact area, which tries to avoid visiting "almost identical" interfaces several times. The kludge was to "uncapture" all superinterfaces for comparison. It seems, that our invention of "uncapture" (JLS has nothing like this) misses one arm of recursion, believing a capture's wildcard is always uncaptured. This is wrong because it may still have a captured bound, as in capture#2-of ? extends capture#1-of ? Making "uncapture" "deeper" lets us accept this example. Let's see what the test suite thinks of it.
