Created attachment 277530 Compilation error in ExtendedWithATest Hello, In my Java 11 project, I have errors reported in Java Editor which are not reported in the Problem view. These errors are all related to modular dependency being used by Test source in another modular Java project. - The project compile fine using Maven 3.6.0 - Eclipse Problem view does not report any errors. As far as I understood, in Maven, test dependencies are put in the class path rather than in module path. This means that the dependent module does not need to "requires" the test dependency (which is fine, as the `module-info.java` would require a test dependency). My use case is the following: - I have made several JUnit 5 Extension or other classes/method, that I use across my project. - Theses extensions/classes/method are modular and have a `module-info.java`. - The other modules using these extensions/classes/method are also modular, but their `module-info.java` never reference test scoped dependency (eg: stuff like junit, assertj, mockito, ...). The error in Eclipse are shown in the attached image: - Errors only appear in the Editor - It is impossible to use method from class in the module: Eclipse will import the class but won't see the method. For example, in the attached test project SharedClassFromA contains a static method which we can't complete (the class complete, and an import is attached) - There is an "The type [...] is not exported from this module". This problem is also reported by Javac, with -Xlint. The attached project ignore this warning because I think it is irrelevant for test source. My expected behavior is no error in the Editor: the module stuff should be ignored for test dependency. For test source, the warning should not be displayed. Note: I've used the m2e category because I don't know where to start: m2e is converting/configuring my pom into Eclipse and is doing whatever it needs with the Note: https://bugs.eclipse.org/bugs/show_bug.cgi?id=535241 is also a bug of mine, and is about the same problem. Don't know how to close/duplicate it. --- Attached files: - module-a-error.png: the error shown by Eclipse as well as Quick Fixe - m2e-modular-test-dependency.zip: a sample project; may be compiled using ./mvn clean install. It can also be imported in Eclipse.

The problem is that NameLookup.getModuleDescription always returns a moduledescriptor for sourcefolders, if there is one in the same project, even if that project is on the classpath (not the modulepath) of the "root" project containing the file in the editor. Once Gerrit is running again, I'll upload a patch that skips looking for a moduledescriptor if the packagefragmentroot if not from the root project and is not on the module path.
