Dev Notes for the patch: The control flow given in your example unearthed an issue where the resources were untracked in TryStatement.analyseCode(). This worked well for �normal� resources which are LocalDeclarations, but for effectively final variables, the scope does not end within twr (or its immediate outer scope for the academic) and hence the warning was not shown (there was no tracking variable since it was withdrawn). Patch corrects this by withdrawing the tracking variable(s) only for LocalDeclaration(s) and then recording that the close was seen (CLOSE_SEEN). However, this causes the ExplicitlyClosed error to be reported which was solved by flagging that this is a FakeTrackingVariable.TWR_EFFECTIVE_FINAL and then checking for the non-existence of this flag at the reporting time. [Could have thought of reusing an existing flag but a new explicit flag gives more code clarity]

Continuing on an observation in the gerrit (the one in TryStatement), I'm not 100% sure where we stand wrt explicit close in the context of this bug. For now I would suggest just to add a couple of tests with explicit close calls, like, e.g.,: Y y = new Y(); if (b) { try (y) {} } else { y.close(); } Also, tests combining "effectively-final-in-twr" with other sources of doubt: variable passed in as method argument, variable passed as arg to other method, as return value of current value. Goal would be to show that these combinations report at the correct level of (un)certainty. I don't expect such tests to reveal a bug, but they would help to precisely capture the state reached. Day-dreaming, the change made me think of more added value: analysis of calling close() multiple times, or in the wrong location. try(y) {} y.close(); // unnecessary or: y.close(); y.read(); // bug Such additional analysis seems feasible based on what we have, but still it would be a new project of its own, which would first of all need an owner.
