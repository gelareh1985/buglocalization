I agree that back support is pain and I'm reconsidering if its worth while spending time on it although I found a work around for it. Reference: docs.oracle.com/javase/9/docs/specs/jar/jar.html Use "automatic-modules" which will work with Java 8, like so: Compile your code to Java 8 target, add the "META-INFO/main/Automatic-Module-Name: org.easylibs.test" attribute (in my test project case) to manifest and build the the jar file. JRE 8 will ignore any META-INFO attributes it doesn't understand, and under Java 9 module-path (not the classpath), it will be treated as an automatic-module. Since we give it that META attribute, it will even have the right module name we want. Jar 9 format also has support for multi-release jar files (JEP 238), but that doesn't work for Java 8 deployments, only 9 or above. However, something to watch for in the future for Eclipse I hope. In any case I am glad you could reproduce that nasty error. Not a very standard configuration, but anytime a messy recursion like that happens, it will always/typically lead to even worse things down the line.

(In reply to Eclipse Genie from comment #8) > New Gerrit change created: https://git.eclipse.org/r/161525 It turned out, that the bug only occurs when *switching* from source 9 to 1.8, i.e., only after the project had already been compiled at source level 9. This resulted in a cached IModuleDescription. Then, when compiling TestApp, the compiler checked if this is associated to a module. A module was found, the module was entered into compilation, but at source 1.8 no ModuleDeclaration could be parsed, so the compiler assumed it must be a regular compilation unit, and asked to which module it is associated. Repeat. The change in gerrit fixes both issues, even though one fix would be enough: When switching source from 9+ to 1.8- we forget the module. When compiling a compilation unit not containing a ModuleDeclaration, we search for the module only if source level is 9+.
