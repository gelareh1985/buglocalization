Creation Time 2019-06-20T06:41:54Z
We have some ARTs running into obscure and rare problems, where some types are not found by JDT. After a lot of debugging, we have observed that the following sequence of events causes the problem: 1. An xtext based job asks whether an annotation (BinaryType) exists via JavaElement.exists(). 2. Our test waits for jobs, one of those jobs being opening a source file. The source file imports the same annotation. 3. The xtext job finds no info for the annotation, as none has been added yet, and enters JavaElement.openWhenClosed(). This xtext job then enters SourceRefElement.generateInfos(). 4. The source file opening job adds the annotation info and and its parent info to the JavaModelManager cache. 5. The xtext job asks for the info of the annotations parent (the class file), which is now present in the JavaModelManager cache. The info is found, so SourceRefElement.generateInfos() does nothing. JavaElement.openWhenClosed() sees that the temporary cache of JavaModelManager does not contain the new type info (although the info has already been added to the non-temporary cache) and throws an exception. 6. BinaryType.exists() returns "false" due to the thrown exception. 7. Our ART fails sporadically, since this situation is not expected by xtext and a chain of hard-to-debug and harder-to-reproduce problems occur. In particular, when the problem occurs during org.eclipse.xtext.common.types.access.jdt.JdtTypeProvider.createResource(URI, IType), there is no JdtTypeMirror attached for the resource and later on we cannot resolve URIs via xtext org.eclipse.xtext.common.types.access.TypeResource.getURIFragment(EObject). It took us days to track this down to the JavaModelCache, since xtext would simply report missing types on the JDT classpath... We see this problem with our product based both on 4.10 and 4.12 official releases. This seems like a bug in SourceRefElement during JavaElement.openWhenClosed(); the code could ask the JavaModelManager for the info of the type once again, if it was not added to the temporary cache by SourceRefElement.generateInfos().

Thanks Simeon for the effort to record the information about the problem in detail in gerrit. Really appreciate going the extra mile! Reproducing that text here in the bug as a one-stop place for all info from https://git.eclipse.org/c/jdt/eclipse.jdt.core.git/commit/?id=ef0b1ee7a4ba561e8d3b59ac4f29dfada60a9cee Concurrent calls to BinaryType.exists() can sometimes result in false for one of the callers, despite the type existing. This is the case due to the two-cache approach in JavaModelManager - it has thread local JavaModelManager.temporaryCache and global shared JavaModelManager.cache. Given two threads calling BinaryType.exists() and nothing is in the global cache, the 1st thread adds the necessary types to the shared JavaModelManager.cache via JavaElement.openWhenClosed(), while the 2nd thread enters JavaElement.openWhenClosed() and is at the beginning of SourceRefElement.generateInfos(). In the SourceRefElement.generateInfos() 2nd thread then finds element info for the parent of the BinaryType and assumes there is nothing to be done, so it leaves the local JavaModelManager.temporaryCache empty. After the call of SourceRefElement.generateInfos() JavaElement.openWhenClosed() can't find anything in the local cache and throws a JavaModelException which causes BinaryType.exists() to return "false", which is wrong and not expected by the caller. With this change, JavaElement.openWhenClosed() takes this case into account and double checks with the JavaModelManager if no info was generated during JavaElement.generateInfos(). This ensures that no exception is thrown and that the correct element info is returned by JavaElement.openWhenClosed(). Without the exception, BinaryType.exists() returns true as expected. This change also adds a test which can sometimes show the behavior with sufficient repetition, if no fix is applied. To observe the problem manually: 1. set repetition in the test to 1 2. set a breakpoint at the start of: SourceRefElement.generateInfos() 3. run the test 4. resume one of the threads at the breakpoint from 2. 5. resume the other thread at the breakpoint from 2. 6. observe that BinaryType.exists() returns false for the 2nd thread Verified for Eclipse 4.13 M1 with Build id: I20190709-1800
