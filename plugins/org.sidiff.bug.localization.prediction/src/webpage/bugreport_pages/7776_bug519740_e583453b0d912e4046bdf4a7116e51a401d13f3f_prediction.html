With the check "Potential resource leak" turned on, the compiler reports that the code below has a problem. If the "while" loop is removed, the problem disappears. class Snippet { static void foo() throws Exception { try (java.util.Scanner scanner = new java.util.Scanner(new java.io.FileInputStream("abc"))) { while (scanner.hasNext()) if (scanner.hasNextInt()) throw new RuntimeException(); /* Potential resource leak: 'scanner' may not be closed at this location */ } } }

(In reply to Eclipse Genie from comment #2) > New Gerrit change created: https://git.eclipse.org/r/155053 The bug was a failure to link the tracking vars for "new Scanner" and "new FileInputStream". As a result, when detecting that "scanner" is managed by try-with-resources we only withdraw the outer tracker, but fail to withdraw also the tracker for the inner FileInputStream. By fixing #innerTracker linkage, all trackers are correctly withdrawn and nothing remains to complain against.
